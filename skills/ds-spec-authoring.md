# DS Spec Authoring

Produces a compact `{component}.spec.json` + `{component}.spec.notes.md` from a Kido DS component.

The spec is the foundation everything else depends on. Accuracy matters more than speed here.

---

## Authoring Process

### Step 1 — Identify the Component

Ask for:
- Component name (as it appears in Kido DS)
- Figma node URL (default file: `https://www.figma.com/design/WNOcZbybSn7dAGjTMimIsJ/Kido---DS`)

### Step 2 — Extract from Figma

Call in this order:

1. `get_metadata` — confirm node, get variant names and IDs
2. `get_design_context` — layer structure, auto-layout, spacing, typography
3. `get_variable_defs` — all design tokens referenced by the component

**Capture:**
- All variant property names and their values
- Layer names (for anatomy)
- Auto-layout direction, gap
- PaddingX, paddingY, min-height, min-width per size
- Typography tokens per size (size, weight, lineHeight)
- Border radius token and value
- All color/opacity tokens — record as a flat `token-name: value` map

### Step 3 — Enrich

After extraction, fill in what Figma doesn't tell you. This is lightweight — answer these and move on:

- Which variant combinations are required vs. optional?
- Does `inverse` exist? Is it generated by default or on request?
- Any known deviations from WCAG (e.g. touch target below 44px)?
- Any design decisions that aren't obvious from the token names?

Do not block on uncertainty. If something is unclear, note it in `spec.notes` or the notes file and continue.

### Step 4 — Write the Spec Files

Write two files:

**`specs/{component-name}.spec.json`** — compact, LLM-optimized. Follow the schema below.

**`specs/{component-name}.spec.notes.md`** — human-readable. Include:
- Non-obvious design decisions
- Token derivation patterns for client brand colors
- Known gaps or deviations
- Version history

### Step 5 — Update the Index

After writing the spec, update `specs/_index.json`. The index is derived from the spec — populate it from the spec's own fields:

```json
{
  "file": "specs/{component-name}.spec.json",
  "component": "{component name from spec}",
  "aliases": [],
  "category": "actions | forms | feedback | navigation | layout | data-display",
  "description": "One sentence — behavior and structure, written for semantic search",
  "anatomy_keywords": ["layer names from spec anatomy + structural synonyms"],
  "related": ["structurally similar components"]
}
```

Update `last_updated` in the index.

### Step 6 — Review

Read the spec back as a plain summary:

```
Button spec:
  Variants: 3 types × 3 sizes × 6 states × inverse (on request) = 54 default / 108 full
  Tokens: 27 resolved
  Sizing: s=32px, m=40px, l=48px
  Notes: size=s below WCAG touch target
```

Ask: "Does this look right? Anything wrong or missing?"

## Spec Schema

```json
{
  "component": "ComponentName",
  "version": "1.0.0",
  "last_updated": "YYYY-MM-DD",
  "source": {
    "figma_file": "URL",
    "figma_node_id": "node:id"
  },

  "variants": {
    "AxisName": {
      "values": ["Value1", "Value2"],
      "default": "Value1",
      "generate_by_default": true
    }
  },

  "naming": {
    "component_set": "lowercase-name-as-in-figma",
    "variant_format": "axis={value}, axis={value}"
  },

  "anatomy": {
    "layers": ["LayerName"],
    "layout": "horizontal | vertical, alignment",
    "gap": 0,
    "notes": "anything not obvious from layer names"
  },

  "sizing": {
    "SizeName": {
      "height": 0, "min_width": 0,
      "paddingX": 0, "paddingY": 0
    }
  },

  "typography": {
    "SizeName": { "token": "token-name", "size": 0, "weight": 0, "lineHeight": 0 }
  },

  "border_radius": { "token": "token-name", "value": 0 },
  "border_width":  { "default": 0, "focus": 0 },

  "tokens": {
    "token/name": "#hexvalue"
  },

  "accessibility": {
    "min_touch_target": 44,
    "focus_ring": { "width": 0, "color_token": "token-name" },
    "contrast": "WCAG AA 4.5:1",
    "aria": "ARIA role and attributes"
  },

  "completeness": {
    "default_set": "human-readable description of default generation scope",
    "full_set": "human-readable description of full generation scope",
    "validation": [
      "plain-English assertion to check after generation"
    ]
  }
}
```

## Spec Maintenance (Updating an Existing Spec)

1. Re-run `get_design_context` and `get_variable_defs` for the component
2. Diff the new extraction against the existing spec — what changed?
3. Update only the changed fields
4. Bump version: patch for token value changes, minor for new variant values, major for structural changes
5. Update `last_updated`
6. Add a history entry to the notes file

Do not rewrite from scratch. Preserve the notes file — it contains context the extraction won't recover.

---

## Spec Health Check

Run this when Kido DS may have changed or a spec feels stale.

```
/ds-spec-authoring check [component-name]
```

1. Pull fresh `get_design_context` and `get_variable_defs` for the component
2. Diff against the existing spec:
   - New or removed variant values?
   - Token values changed?
   - New layers in anatomy?
3. Report: "3 token values changed, 1 new state added, sizing unchanged"
4. Ask: "Update the spec?"

---

## Fallback (MCP Unavailable)

Ask the user to paste the component's variant properties and layer structure manually.
A screenshot is acceptable — describe what you see and ask the user to confirm.
Note the fallback method in the spec's source field: `"figma_node_id": "unknown — extracted manually"`.
